# BMad 常见问题：菜单之外的任务

**版本**: 1.0
**日期**: 2025-11-19

---

## Q: 菜单里没有的任务（比如 "帮我修复 xx 错误"），是不是就不适合用 BMad Agent？

**A: 并不是不适合，而是有 "快捷方式" 和 "标准方式" 的区别。**

BMad 的菜单设计是为了覆盖 **最常见、最需要结构化** 的工作流（如开发新功能、设计架构）。但 Agent 本身是基于大语言模型的专家，完全有能力处理菜单之外的临时任务。

### 1. 快捷方式 (Ad-hoc Mode)

你可以直接用自然语言向 Agent 提问或下达指令。

- **场景**:
  - "帮我解释一下这段代码。"
  - "修复 `src/utils.ts` 里的类型错误。"
  - "给这个函数写个单元测试。"
  - "这个 SQL 查询怎么优化？"

- **如何操作**:
  - 直接呼叫对应的专家 Agent (例如 `@bmm-dev` 或 `@bmm-architect`)。
  - 清晰描述你的问题，最好附上相关文件路径。
  - Agent 会利用其角色知识直接回答或执行。

- **优点**: 快速，灵活。
- **缺点**: 缺乏上下文的深度验证，可能不如完整工作流严谨。

### 2. 标准方式 (Workflow Mode)

对于复杂的 "错误修复" 或 "优化"，建议将其视为一个 **微型任务 (Story)**。

- **场景**:
  - "修复生产环境的内存泄漏问题。" (这通常不是改一行代码就能解决的)
  - "重构整个认证模块。"
  - "优化首页加载速度。"

- **如何操作**:
  1.  呼叫 `@bmm-sm` 运行 `*create-story`，创建一个名为 "Fix Memory Leak" 的 Story。
  2.  在 Story 中描述复现步骤和预期结果。
  3.  运行 `*story-context` 让 Agent 自动收集相关代码。
  4.  呼叫 `@bmm-dev` 运行 `*develop-story`。

- **优点**:
  - **上下文完整**: Agent 会自动分析依赖关系。
  - **测试驱动**: 流程会强制要求编写/运行测试，确保修复有效且不引入新 Bug。
  - **记录在案**: 你的修复过程会被记录在文档中。

### 总结

- **小修小补 / 咨询**: 直接对话，无需菜单。
- **复杂修复 / 功能开发**: 请走菜单流程 (Story -> Develop)，这是 BMad 发挥最大威力的 "正道"。

**记住**: 菜单是 "最佳实践" 的集合，而不是能力的边界。
